## 背景与目标

在 vLLM 场景下，显存占用可粗分为四类：模型权重、KV Cache、中间激活、部分算子持久缓存。本文不讨论算子持久缓存，仅聚焦前 3 类。

其中模型权重和 KV Cache 更偏“长期占用”，中间激活更偏“瞬时占用”。`torch_memory_saver` 的核心价值是：在不改变虚拟地址的前提下，临时释放对应显存并在需要时恢复，以提升阶段性可用显存。

本文目标不是实现代码，而是明确：
1. 基于现有框架，哪些能力可直接支持。
2. 哪些需求需要改 API 或改 C++ 内核。
3. 推荐的演进路径与风险控制方式。

## 现有框架能力与边界

### 1) 内存管理机制

- 当前实现基于 CUDA VMM：
  - 分配时：`cuMemCreate + cuMemAddressReserve + cuMemMap`。
  - 暂停时：`cuMemUnmap + cuMemRelease`，保留虚拟地址。
  - 恢复时：重新 `cuMemCreate + cuMemMap` 到原地址。
- 结果：
  - 地址稳定，天然适配 CUDA Graph 场景。
  - 释放的是物理显存，不是单纯“逻辑标记”。

### 2) tag 与 mem_pool 关系

- Python 侧 `region(tag=..., enable_cpu_backup=...)` 使用 `(tag, enable_cpu_backup)` 作为 mem_pool key。
- C++ 侧 metadata 记录每个 allocation 的 `tag`、`state`、`enable_cpu_backup` 等。
- `pause(tag)` / `resume(tag)` 当前是 **tag 精确匹配**，`tag=None` 表示作用于全部受管 allocation。

### 3) 状态机约束（非常关键）

- allocation 状态只有 `ACTIVE` / `PAUSED`。
- 对非 `ACTIVE` allocation 调用 `pause`，或对非 `PAUSED` allocation 调用 `resume`，当前实现会直接报错并退出。
- 这意味着：上层必须保证调用序列严格正确，不允许“重复 pause/resume”。
- 设计原则：重复 `pause/resume` 视为上层编排错误，底层不做容错去重，仅做快速失败并暴露错误现场（tag/ptr）。

### 4) CPU backup 当前语义

- 当前 `enable_cpu_backup=True` 的行为是“瞬时备份”：
  - `pause` 时分配 pinned host 内存并 D2H 拷贝。
  - `resume` 时 H2D 拷回并释放 host 备份。
- 适合会变化的普通张量；对“长期不变权重”会产生重复分配/拷贝成本。

## 需求 1：层级化 tag 设计

### 需求描述

将 tag 扩展为层级形式（如 `model.layer0`、`model.layer1`），希望支持：
- `pause("model", recursive=True)`：作用于 `model.*`。
- `pause("model", sub_tags=["layer0","layer2"])`：拼接字符并作用于子集。
- `recursive=False` 时仍保持精确匹配语义。
- 补充说明：该层级是“逻辑分组”，不是新的物理池类型；底层仍按实际 tag 归属的 mem_pool 管理。

### 可行性评估

结论：**可行，推荐渐进式落地**。

- 现有结构已具备核心要素（每个 allocation 已记录字符串 tag）。
- 主要缺口是“匹配策略与 API 表达能力”而非底层 VMM 能力。
- 不建议在底层做“父子重叠去重并吞错”。当前严格状态机有价值，重复操作应视为上层编排错误并显式失败。

### 推荐方案

方案 A：在 C++ pause/resume 中新增匹配模式
- 思路：扩展 `tms_pause/tms_resume` 参数，支持 `exact/prefix/list` 等匹配模式。
- 优点：
  - 语义集中在单一真源（C++ allocation metadata），不依赖 Python 侧“已知 tag 集合”。
  - 对 preload / torch 两种 hook mode 一致。
- 成本：
  - 需改 C API、ctypes 签名与 Python 封装。



## 需求 2：权重场景的“预置 backup（eager/persistent）”

### 需求描述

对于模型权重这类“内容稳定”的张量，期望 CPU backup 不再采用“每次 pause 创建、每次 resume 释放”的瞬时模式，而是支持：
- 在合适时机（通常为权重加载完成后）建立 host 备份；
- pause/resume 周期内复用备份；
- 仅在显式销毁或对象释放时回收备份。

### 可行性评估

结论：**可行，但需要增加 backup 生命周期状态**。

现有实现的问题不在能力缺失，而在策略单一：
- 当前 metadata 只有 `enable_cpu_backup(bool)`，表达不了“瞬时”与“持久”差异。
- 需要至少新增 backup policy（例如 `none/ephemeral/persistent`）和备份初始化状态。

### 新增 metadata 设计

为避免通过 tag 名称约定来区分策略，需在 allocation metadata 中新增（或扩展）以下字段：
- `cpu_backup_policy`：`none | ephemeral | persistent`。
- `backup_ready`：当前 backup 是否可用于恢复（尤其用于异步 refresh 场景）。
- `backup_inflight`：是否存在进行中的异步 backup 传输。
- `backup_generation`（可选）：backup 版本号，用于上层排查“是否用了陈旧备份”。

字段职责说明：
- `tag` 仅负责分组与匹配范围，不承载 backup 生命周期语义。
- `cpu_backup_policy` 才是区分 `ephemeral/persistent` 的唯一策略字段。

### 建议语义

- `ephemeral`（兼容当前默认）：
  - pause 时创建/更新备份；resume 后释放备份。
- `persistent`（面向不可变权重）：
  - 不在首次分配时自动建立备份（此时权重可能尚未加载完成，备份可能是脏数据）；
  - 由上层在“权重加载完成后”显式触发备份建立/刷新；
  - 后续 pause/resume 复用同一备份，不在 resume 释放；
  - 支持主动释放 backup；若未主动释放，则由 free 路径自动回收。

异步备份完成确认：
- 若上层使用异步刷新，底层需要维护完成状态（如 event）。
- `resume(tag, wait_backup=True)` 时，若目标备份未完成则等待；`wait_backup=False` 时不等待并在未就绪时快速失败。
- 可选提供只读查询接口（如 `is_backup_ready(tag)`）用于上层编排。

注意：`persistent` 必须配合“上层保证数据不变”或提供“脏标记刷新机制”，否则会恢复陈旧数据。
补充约定：该语义正确性由上层保证。底层仅提供机制，不负责判断业务语义是否正确；因此需要在 `persistent` 接口说明中显式标红“误用会恢复陈旧数据”。

## 推荐 API 草案（供讨论）

保持向后兼容的前提下，建议在 Python 层新增：

```python
torch_memory_saver.pause(
    tag: str | None = None,
    recursive: bool = False,
    sub_tags: list[str] | None = None,
)

torch_memory_saver.resume(
    tag: str | None = None,
    recursive: bool = False,
    sub_tags: list[str] | None = None,
    wait_backup: bool = True,
)

with torch_memory_saver.region(
    tag: str = "default",
    cpu_backup_policy: str = "none",  # none | ephemeral | persistent
):
    ...

torch_memory_saver.refresh_persistent_backup(
    tag: str,
    async_copy: bool = False,
)

torch_memory_saver.release_persistent_backup(
    tag: str,
)
```

说明：以上 API 面向普通张量/模型权重的“整段”管理，不覆盖 KV Cache 的区间级管理。

兼容策略：
- 旧参数 `enable_cpu_backup=False/True` 可映射为 `none/ephemeral`。
- 未指定新参数时保持当前行为不变。
- C++ 侧 allocation metadata 需要落地 `cpu_backup_policy` 等字段，禁止通过 tag 字符串约定推断策略。
- `release_persistent_backup(tag)` 仅允许作用于 `cpu_backup_policy=persistent` 的 allocation；若匹配到 `ephemeral/none`，应快速失败并报错。

## 风险与规避

1. 状态机冲突风险
- 风险：层级 tag 批量操作容易产生重复 pause/resume。
- 规避：保持“冲突即失败”，并在错误信息中输出冲突 tag 与 ptr。

2. 备份一致性风险
- 风险：`persistent` 用于可变张量会恢复旧值。
- 规避：文档与 API 命名中明确“仅建议不可变数据”；提供 `refresh_persistent_backup(tag, async_copy=...)` 由上层在正确时机刷新。

3. 接口误用风险
- 风险：上层对 `ephemeral/none` 调用主动释放接口，导致语义混乱。
- 规避：`release_persistent_backup(tag)` 仅对 `persistent` 生效，其他策略一律快速失败。

4. 内存占用转移风险
- 风险：persistent 会把压力转移到 host 内存（通常是 pinned memory）。
- 当前阶段策略：沿用现有 host backup 机制，不新增 pinned/async 优化路径；相关性能优化留到后续阶段。
- 当前阶段接口：提供 `release_persistent_backup(tag)` 主动回收；同时保留 allocation `free` 路径自动回收。
- 后续演进：若线上观测到 host 内存峰值压力，再补按 tag/总量的统计与显式回收能力。

## 需求 3：KV Cache 专项设计

### 3.1 KV Cache 与普通张量的差异

- 普通张量/模型权重的 pause/resume 以“整段内存”为主。
- KV Cache 在 vLLM 中表现为“整体预留、局部访问”，因此更适合“局部 pause / 局部 resume”模型。
- KV Cache 多数场景依赖重计算或失效重建，通常不要求 CPU backup。

### 3.2 KV Cache 的 VMM 映射思路（草案）

初始化阶段：
- 为 KV Cache 的整段虚拟地址空间完成地址保留与访问权限设置。
- 将整段地址先映射到“占位句柄”（可理解为空页占位）以保证地址空间连续且可管理。

局部 `resume` 阶段：
- 按上层请求的区间大小创建真实句柄。
- 对目标区间执行 `unmap(占位) -> map(真实句柄) -> set access`。

局部 `pause` 阶段：
- 对目标区间执行 `unmap(真实句柄) -> release(真实句柄) -> map(占位句柄) -> set access`。

实现约束：
- 底层按上层给定区间执行，不负责推断“应该映射多大”。
- 若上层给出非法区间或不对齐区间，底层快速失败并报错。

### 3.3 与现有框架的一致性评估

- 当前框架已支持“整段 pause/resume”与 tag 维度管理。
- KV Cache 方案需要新增“区间级 map/unmap/release”能力与区间元数据管理，属于新增能力，不是现有接口的小改。
- 建议把 KV Cache 作为独立子特性推进，避免与 `persistent backup` / 层级 tag 改造耦合。

### 3.4 API 设计结论

- 结论：KV Cache 采用独立 API，不在现有 `pause/resume(tag)` 上追加区间参数。
- 原因：
  - 现有 API 语义是整段 allocation；KV Cache 语义是区间级 allocation，抽象层级不同。
  - 若强行复用，将显著增加参数复杂度并污染普通张量路径。

建议的 KV Cache API 形态（草案）：
- `kvcache_register(tag, base_ptr, total_size, block_size, ...)`
- `kvcache_resume_ranges(tag, ranges, ...)`
- `kvcache_pause_ranges(tag, ranges, ...)`
- `kvcache_release(tag)`（释放对应 KV Cache 管理结构）
